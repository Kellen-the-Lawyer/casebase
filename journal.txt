## Future work — AUTOMATED EXPERIENCE COMPARISON (TODO)

Implement automated total-time comparison against PWD requirements:
  - Sum saved months across all letters and compare against the years required by the PWD
  - This is non-trivial because PERM language often phrases requirements as "3 years of
    experience in the field" but lists skills (Python, HTML, CSS) that appear to require
    3 years each — when in practice only 3 total years in the field is required and the
    skills just need to appear somewhere in the letters
  - Any automated comparison must account for this language distinction to avoid false
    flags; suggest a two-pass approach: (1) flag if total time is below threshold, (2)
    separately verify each listed skill appears in at least one letter, without asserting
    time-per-skill
  - Consider adding a "time goal" input so users can set the target years manually rather
    than trying to parse it from the PWD text

## Completed this session

### Removed all AI/Anthropic API calls from the frontend

Client consent and internal approval would be required before sending any data to
a third-party AI API. All four AI-powered features have been replaced with
server-side pdfplumber + regex processing. No data leaves the local backend.

### New file: extract_experience_letter.py

Rule-based pdfplumber extractor for experience verification letters. Extracts:
  - employerName    — first letterhead line
  - employerTitle   — signer's title from signature block
  - beneficiaryName — name found in employment sentence
  - jobTitle        — parsed from "employed as / position of / served as" triggers
  - startDate / endDate — as written in the letter
  - startParsed / endParsed — normalised to YYYY-MM-DD (ISO)
  - months          — computed duration
  - duties          — body paragraph(s) after duties/responsibilities heading
  - skills          — tech/professional keywords matched against a curated list
  - fullText        — raw extracted text

Handles named months, MM/YYYY, MM/DD/YYYY, YYYY-MM formats.
Recognises "Present" / "current" / "ongoing" as end date.

### New API endpoints in api.py

  POST /api/extract-text
    Generic PDF → plain text via pdfplumber. Used by drag-drop boxes.

  POST /api/extract-experience-letter
    Structured letter parsing via extract_experience_letter.py.
    Used by the Verify Experience modal upload.

### Frontend changes

  DropTextarea (Job Description boxes):
    - Previously called /api/claude (Anthropic) with base64 PDF
    - Now POSTs multipart form to /api/extract-text (pdfplumber, local)

  ExperienceVerificationModal (letter upload):
    - Previously called /api/claude with base64 PDF + AI prompt
    - Now POSTs multipart form to /api/extract-experience-letter (pdfplumber, local)

  "Analyze vs PWD" button removed entirely.
    - This feature requires AI reasoning and cannot be replaced with rule-based logic
    - Noted in Session 8 TODO for future implementation once AI use is approved

  Removed unused client-side helpers: parseFlexDate, dateDiffMonths
    (date math now done in Python backend and returned as ISO strings + integer months)

  /api/claude proxy endpoint retained in api.py but no longer called from the frontend.

Root cause: pdfplumber outputs form table cells differently depending on the PDF
renderer. Three layouts were observed and are now handled:

  Layout A (most common in new form): labels row + values row
    "3. City *    4. State *    5. County *\nRichardson    TX    Dallas County"

  Layout B: each label on its own line, value on next line
    "3. City *\nRichardson\n4. State *\nTX"
    Already worked; kept as fallback.

  Layout C: label and value on same line (table cell)
    "3. City *    Richardson"
    Added as second fallback.

Additional fixes in _extract_new:
  - Job title: now handles same-line placement ("1. Job title *   Data Scientist II")
    in addition to next-line placement
  - All searches now use body = pages[1:7] instead of hard-coded page indices,

  Passing `"2026-01-01"` caused a 500 crash whenever date-only searches were used.

- Added `try/catch/finally` to both frontend search functions so the spinner
  always clears and errors surface in the UI instead of hanging forever.
