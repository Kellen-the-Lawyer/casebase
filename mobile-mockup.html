Design Proposal · Revised

Casebase Mobile Web

Browser-appropriate phone layout — no bottom nav, no native app patterns. Uses a sticky top header with a hamburger drawer, proper browser chrome clearance, and the browser's own back button for navigation.

↑
What changed from the previous mockup: Bottom nav is gone entirely. Each phone shell now shows a simulated Safari address bar at the top — this is the space the browser owns. The app header sits below that. Navigation lives in a slide-out left drawer triggered by the hamburger icon. The browser's own back/forward buttons (shown in the Safari toolbar) handle navigation history, which requires moving from the current state-based router to proper URL routing.
Screens 1–3
Home, Search, Document Reader

The top header is sticky but slim (48px). The hamburger opens the left drawer. Search and result cards take the full width. The document reader is a single-pane view — no split panel.

Summary
What makes this web-appropriate
Navigation
  • Hamburger + left drawer — browser standard, no conflict with Safari toolbar
  • Browser back/forward button handles history — no in-app back arrows needed
  • All 12 sections visible in drawer with grouping — nothing buried
URL Routing (required change)
  • Current app uses state-based routing — browser back does nothing
  • Must move to real URLs (React Router or similar) for mobile web to feel correct
  • Bonus: pages become shareable links, work with browser bookmarks
Browser Chrome Awareness
  • Safari toolbar shown at bottom — no fixed elements compete with it
  • App scrolls freely; only the 48px app header is sticky
  • Chat input uses position:sticky bottom, not fixed, so keyboard push works correctly
Features Adapted
  • Force graph → citation list (your call — keeping it)
  • Pivot builder → desktop only, 4 KPIs + mini bars on mobile
  • Sparklines → movement delta cards (plain text, same info)
  • Split-pane document view → full-width single pane
